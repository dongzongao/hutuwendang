# 三色标记算法详解

三色标记是 JVM 并发垃圾回收（CMS、G1、ZGC）的核心算法，解决了 GC 线程与用户线程并发运行时的对象漏标问题。

---

## 三种颜色含义

| 颜色 | 含义 |
|------|------|
| 白色 | 未被 GC 访问，初始状态。GC 结束后仍是白色 → 垃圾，回收 |
| 灰色 | 已被 GC 访问，但其引用的子对象还未全部扫描完 |
| 黑色 | 已被 GC 访问，且其所有子对象都已扫描完，不会被回收 |

---

## 标记过程

```
初始：所有对象白色

1. GC Roots 直接引用的对象 → 灰色
   GC Roots → [A灰] → [B白] [C白]

2. 扫描灰色对象 A 的子对象，子对象变灰，A 变黑
   GC Roots → [A黑] → [B灰] [C灰]

3. 继续扫描 B、C，无子对象，变黑
   GC Roots → [A黑] → [B黑] [C黑]

4. 无灰色对象，标记结束
   白色对象 → 回收
```

---

## 并发标记的问题

GC 线程标记的同时，用户线程也在修改对象引用，会出现两种问题：

### 多标（浮动垃圾）

```
GC 扫描完 A（黑色），用户线程删除 A→B 的引用
B 已经没有引用了，但本轮 GC 不会回收它
→ B 成为浮动垃圾，下次 GC 才回收，可以接受
```

### 漏标（致命问题）

```
GC 正在扫描，A 是黑色，D 是白色
用户线程：A 新增引用 → D，同时删除 B → D 的引用

结果：
- A 已经是黑色，不会再扫描
- D 的唯一引用路径断了（B→D 被删）
- D 是白色 → 被当成垃圾回收！但 D 还活着！
```

漏标的触发条件（需同时满足）：
1. 黑色对象新增了对白色对象的引用
2. 所有灰色对象到该白色对象的引用被删除

---

## 解决漏标：两种屏障

### 增量更新（Incremental Update）— CMS 使用

黑色对象新增引用时，将该黑色对象重新标记为灰色，下次重新扫描。

```
A（黑）新增 → D（白）
→ A 重新变灰，下轮重新扫描 A 的所有引用
```

### 原始快照（SATB，Snapshot At The Beginning）— G1、ZGC 使用

灰色对象删除对白色对象的引用时，将被删除的引用记录下来，GC 结束前重新扫描这些引用。

```
B（灰）删除 → D（白）
→ 记录 D，GC 结束前把 D 当作灰色重新扫描
→ D 可能是浮动垃圾，但不会漏标
```

---

## 两种方案对比

| 方案 | 使用者 | 触发时机 | 特点 |
|------|--------|----------|------|
| 增量更新 | CMS | 新增引用时 | 重新扫描黑色对象，重新标记 STW 时间较长 |
| SATB | G1、ZGC | 删除引用时 | 保留快照，可能产生浮动垃圾，并发效率更高 |

---

## 在各 GC 中的体现

### CMS

使用增量更新 + 写屏障，在引用新增时记录变更，并发标记结束后有一次重新标记（STW）阶段，扫描增量更新记录的对象。

### G1

使用 SATB + 写屏障，在引用被覆盖时触发，将旧引用值记录到 SATB 队列，并发标记结束后统一重新扫描队列中的对象。

### ZGC

更进一步，用读屏障（Load Barrier）替代写屏障，在对象被读取时检查颜色，配合染色指针（Colored Pointers）实现几乎全程并发，STW 时间控制在 1ms 以内。

```
染色指针：将 GC 状态信息直接编码在指针的高位 bit 中
读屏障：每次读取对象引用时，检查指针颜色，触发必要的修正操作
→ 不需要扫描整个堆来找变更，效率极高
```
