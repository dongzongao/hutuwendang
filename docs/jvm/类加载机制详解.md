# JDK 类加载 与 Spring Boot 类加载详解

## 一、JDK 类加载机制

### 1. 类的生命周期

```
类的完整生命周期：

加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
       ├──────────────────┤
              连接阶段

┌──────────────────────────────────────────────────────────┐
│  加载（Loading）                                          │
│  - 通过类名获取 .class 文件的二进制字节流                  │
│  - 将字节流转化为方法区的运行时数据结构                    │
│  - 在堆中生成 java.lang.Class 对象                        │
├──────────────────────────────────────────────────────────┤
│  验证（Verification）                                     │
│  - 文件格式验证：魔数 0xCAFEBABE、版本号                  │
│  - 元数据验证：语义分析，是否有父类等                      │
│  - 字节码验证：数据流和控制流分析                          │
│  - 符号引用验证：解析阶段前的最后校验                      │
├──────────────────────────────────────────────────────────┤
│  准备（Preparation）                                      │
│  - 为类变量（static）分配内存                             │
│  - 设置类变量的初始零值（不是代码中的初始值）              │
│  - static final 常量直接赋值                              │
├──────────────────────────────────────────────────────────┤
│  解析（Resolution）                                       │
│  - 将常量池中的符号引用替换为直接引用                      │
│  - 类/接口、字段、方法的解析                               │
├──────────────────────────────────────────────────────────┤
│  初始化（Initialization）                                 │
│  - 执行 <clinit>() 方法（类构造器）                       │
│  - 执行静态变量赋值和静态代码块                            │
│  - 父类的 <clinit>() 先于子类执行                         │
└──────────────────────────────────────────────────────────┘
```

### 2. 触发类初始化的时机（主动引用）

```java
public class InitTrigger {

    /**
     * 以下 6 种情况会触发类初始化：
     *
     * 1. new 实例化对象
     * 2. 访问/修改类的静态变量（非 final 常量）
     * 3. 调用类的静态方法
     * 4. 反射调用
     * 5. 初始化子类时，父类未初始化则先初始化父类
     * 6. JVM 启动时的主类（main 方法所在类）
     */

    static class Parent {
        static int value = 100;
        static {
            System.out.println("Parent 初始化");
        }
    }

    static class Child extends Parent {
        static {
            System.out.println("Child 初始化");
        }
    }

    public static void main(String[] args) {
        // 触发 Parent 初始化，不触发 Child 初始化
        System.out.println(Child.value);

        // 不触发初始化的情况（被动引用）：
        // 1. 通过子类引用父类静态字段
        // 2. 通过数组定义引用类：Child[] arr = new Child[10]
        // 3. 引用类的 static final 常量（编译期已放入常量池）
    }
}
```

### 4. 类加载器

```
JDK 类加载器层次结构：

┌─────────────────────────────────────────────────────┐
│  Bootstrap ClassLoader（启动类加载器）                │
│  - C++ 实现，JVM 内置                                │
│  - 加载 JAVA_HOME/lib 下的核心类库                   │
│  - rt.jar、charsets.jar 等                          │

                       │ 父加载器

│  - JDK9+ 改为 Platform ClassLoader                  │
└──────────────────────┬──────────────────────────────┘
                       │ 父加载器
┌──────────────────────▼──────────────────────────────┐
│  Application ClassLoader（应用类加载器）              │
│  - Java 实现：sun.misc.Launcher$AppClassLoader       │
│  - 加载 classpath 下的类                             │
│  - 默认的类加载器                                    │
└──────────────────────┬──────────────────────────────┘
                       │ 父加载器
┌──────────────────────▼──────────────────────────────┐
│  自定义 ClassLoader                                  │
│  - 继承 ClassLoader 或 URLClassLoader                │

└─────────────────────────────────────────────────────┘
```

### 5. 双亲委派模型

```java
public class ParentDelegation {

    /**
     * 双亲委派工作流程：
     *
     * 1. 收到类加载请求
     * 2. 先委派给父加载器处理
     * 3. 父加载器无法加载时，才由自己加载
     *
     * ┌──────────────────────────────────────────┐
     * │  自定义 ClassLoader                       │
     * │  收到加载 java.lang.String 请求           │
     * │  ↓ 委派给父加载器                         │
     * │  Application ClassLoader                  │
     * │  ↓ 委派给父加载器                         │
     * │  Extension ClassLoader                    │
     * │  ↓ 委派给父加载器                         │
     * │  Bootstrap ClassLoader                    │
     * │  ✓ 在 rt.jar 中找到，加载成功             │
     * └──────────────────────────────────────────┘
     */

    // ClassLoader.loadClass() 源码逻辑
    protected Class<?> loadClass(String name, boolean resolve)
            throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            // 1. 先检查是否已加载
            Class<?> c = findLoadedClass(name);

            if (c == null) {
                try {
                    // 2. 委派给父加载器
                    if (parent != null) {
                        c = parent.loadClass(name, false);
                    } else {
                        // 父加载器为 null，委派给 Bootstrap
                        c = findBootstrapClassOrNull(name);
                    }
                } catch (ClassNotFoundException e) {
                    // 父加载器无法加载，继续
                }

                if (c == null) {
                    // 3. 父加载器失败，自己尝试加载
                    c = findClass(name);
                }
            }

            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }
}
```

**双亲委派的优势**：

```
1. 安全性：
   - 防止核心类被篡改
   - 自定义 java.lang.String 不会被加载
   - 核心类只由 Bootstrap 加载

2. 避免重复加载：
   - 父加载器已加载的类不会重复加载
   - 保证类的唯一性

3. 层次清晰：
   - 类加载有明确的优先级
   - 便于管理和维护
```

### 6. 破坏双亲委派

```java
/**
 * 三种破坏双亲委派的场景：
 *
 * 1. JDK 1.2 之前（历史遗留）
 * 2. SPI 机制（JDBC、JNDI 等）
 * 3. OSGi、热部署等模块化场景
 */

// 场景2：SPI 破坏双亲委派
// 问题：java.sql.Driver 由 Bootstrap 加载
//       但具体实现（mysql-connector）在 classpath
//       Bootstrap 无法加载 classpath 的类

// 解决：线程上下文类加载器（Thread Context ClassLoader）
public class SPIExample {

    public static void main(String[] args) {
        // JDBC 加载驱动的过程
        // DriverManager 由 Bootstrap 加载
        // 但通过线程上下文类加载器加载具体驱动实现

        ClassLoader contextClassLoader =
            Thread.currentThread().getContextClassLoader();
        // 默认是 AppClassLoader，可以加载 classpath 的类

        // ServiceLoader 使用线程上下文类加载器
        ServiceLoader<Driver> drivers =
            ServiceLoader.load(Driver.class);
    }
}

// 自定义类加载器（打破双亲委派）
public class CustomClassLoader extends ClassLoader {

    private String classPath;

    public CustomClassLoader(String classPath) {
        // 不传 parent，打破委派
        super(null);
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] classData = loadClassData(name);
        if (classData == null) {
            throw new ClassNotFoundException(name);
        }
        return defineClass(name, classData, 0, classData.length);
    }

    private byte[] loadClassData(String className) {
        // 从自定义路径加载字节码
        String path = classPath + className.replace('.', '/') + ".class";
        try (InputStream is = new FileInputStream(path)) {
            return is.readAllBytes();
        } catch (IOException e) {
            return null;
        }
    }
}
```

### 7. 类加载器实战

```java
public class ClassLoaderDemo {

    public static void main(String[] args) throws Exception {
        // 获取各层类加载器
        ClassLoader appLoader = ClassLoaderDemo.class.getClassLoader();
        ClassLoader extLoader = appLoader.getParent();
        ClassLoader bootLoader = extLoader.getParent(); // null

        System.out.println("App: " + appLoader);
        // sun.misc.Launcher$AppClassLoader@...
        System.out.println("Ext: " + extLoader);
        // sun.misc.Launcher$ExtClassLoader@...
        System.out.println("Boot: " + bootLoader);
        // null（Bootstrap 是 C++ 实现）

        // String 由 Bootstrap 加载
        System.out.println(String.class.getClassLoader());
        // null

        // 动态加载类
        URL[] urls = {new URL("file:///path/to/classes/")};
        URLClassLoader loader = new URLClassLoader(urls);
        Class<?> clazz = loader.loadClass("com.example.MyClass");
        Object instance = clazz.getDeclaredConstructor().newInstance();

        // 同一个类被不同加载器加载 → 不同的类
        ClassLoader loader1 = new CustomClassLoader("/path1/");
        ClassLoader loader2 = new CustomClassLoader("/path2/");
        Class<?> c1 = loader1.loadClass("com.example.Foo");
        Class<?> c2 = loader2.loadClass("com.example.Foo");
        System.out.println(c1 == c2); // false！
    }
}
```


## 二、Spring Boot 类加载机制

### 1. Spring Boot Fat Jar 结构

```
Spring Boot 打包后的 Fat Jar 结构：

myapp.jar
├── META-INF/
│   └── MANIFEST.MF                    ← 启动入口配置
├── BOOT-INF/
│   ├── classes/                       ← 应用自身的类
│   │   ├── com/example/
│   │   │   └── Application.class
│   │   └── application.yml
│   └── lib/                           ← 所有依赖 jar
│       ├── spring-core-6.0.0.jar
│       ├── spring-boot-3.0.0.jar
│       └── mysql-connector-j-8.0.jar
└── org/springframework/boot/loader/   ← Spring Boot Loader
    ├── JarLauncher.class
    ├── LaunchedURLClassLoader.class
    └── ...

MANIFEST.MF 内容：
Main-Class: org.springframework.boot.loader.JarLauncher
Start-Class: com.example.Application
Spring-Boot-Version: 3.0.0
Spring-Boot-Classes: BOOT-INF/classes/
Spring-Boot-Lib: BOOT-INF/lib/
```

### 2. Spring Boot 启动流程

```java
/**
 * Spring Boot 启动类加载流程：
 *
 * 1. JVM 执行 JarLauncher.main()
 * 2. JarLauncher 创建 LaunchedURLClassLoader
 * 3. LaunchedURLClassLoader 加载 BOOT-INF/classes 和 BOOT-INF/lib
 * 4. 用 LaunchedURLClassLoader 加载并执行 Start-Class（Application）
 */

// JarLauncher 核心逻辑（简化）
public class JarLauncher extends ExecutableArchiveLauncher {

    @Override
    protected boolean isNestedArchive(Archive.Entry entry) {
        // 判断是否是需要加载的嵌套 jar
        if (entry.isDirectory()) {
            return entry.getName().equals("BOOT-INF/classes/");
        }
        return entry.getName().startsWith("BOOT-INF/lib/");
    }

    public static void main(String[] args) throws Exception {
        new JarLauncher().launch(args);
    }
}

// Launcher.launch() 核心逻辑（简化）
protected void launch(String[] args) throws Exception {
    // 1. 注册 URL 协议处理器（支持 jar 中的 jar）
    JarFile.registerUrlProtocolHandler();

    // 2. 创建类加载器
    ClassLoader classLoader = createClassLoader(getClassPathArchivesIterator());

    // 3. 获取 Start-Class
    String launchClass = getMainClass();

    // 4. 用新类加载器启动主类
    launch(args, launchClass, classLoader);
}
```

### 3. LaunchedURLClassLoader

```java
/**
 * LaunchedURLClassLoader 是 Spring Boot 的核心类加载器
 *
 * 继承关系：
 * LaunchedURLClassLoader
 *   └── URLClassLoader
 *         └── SecureClassLoader
 *               └── ClassLoader
 *
 * 职责：
 * - 加载 BOOT-INF/classes/ 下的应用类
 * - 加载 BOOT-INF/lib/ 下所有 jar 中的类
 * - 支持从嵌套 jar 中加载类（jar in jar）
 */
public class LaunchedURLClassLoader extends URLClassLoader {

    /**
     * 类加载顺序（与标准双亲委派略有不同）：
     *
     * 1. 检查是否已加载
     * 2. 委派给 Bootstrap ClassLoader
     * 3. 自己尝试加载（BOOT-INF/classes 和 BOOT-INF/lib）
     * 4. 委派给 AppClassLoader（classpath）
     *
     * 注意：Spring Boot 优先自己加载，再委派给 AppClassLoader
     * 这与标准双亲委派（先委派父加载器）略有不同
     */

    @Override
    protected Class<?> loadClass(String name, boolean resolve)
            throws ClassNotFoundException {
        // 处理特殊包（如 org.springframework.boot.loader）
        // 这些类由 AppClassLoader 加载，不走 LaunchedURLClassLoader
        if (name.startsWith("org.springframework.boot.loader")) {
            try {
                Class<?> result = super.loadClass(name, resolve);
                if (result != null) return result;
            } catch (ClassNotFoundException e) { }
        }
        return super.loadClass(name, resolve);
    }
}
```

### 4. 嵌套 Jar 加载原理

```java
/**
 * 标准 URLClassLoader 无法加载 jar 中的 jar
 * Spring Boot 通过自定义 URL 协议解决这个问题
 *
 * 标准 URL：jar:file:/path/to/app.jar!/com/example/Foo.class
 * Spring Boot URL：jar:file:/path/to/app.jar!/BOOT-INF/lib/dep.jar!/com/example/Bar.class
 *                                                                    ↑ 嵌套 jar 路径
 */

// 注册自定义 URL 协议处理器
public class JarFile {
    public static void registerUrlProtocolHandler() {
        // 注册 "jar" 协议的自定义处理器
        // 支持 jar:file:xxx.jar!/nested.jar!/class 格式
        String handlers = System.getProperty(PROTOCOL_HANDLER, "");
        System.setProperty(PROTOCOL_HANDLER,
            ("".equals(handlers) ? HANDLERS_PACKAGE
                : handlers + "|" + HANDLERS_PACKAGE));
    }
}

// 自定义 JarURLConnection 处理嵌套 jar 的读取
// 核心：将嵌套 jar 的内容映射到内存，再按标准方式读取
```

### 6. Spring Boot DevTools 热重载

```java
/**
 * DevTools 使用两个类加载器实现热重载：
 *
 * RestartClassLoader（重启类加载器）：
 * - 加载开发者自己写的类（classpath 中可变的部分）

 * 6. 重启 Spring ApplicationContext
```

## 四、常见问题与解决

### 1. ClassNotFoundException vs NoClassDefFoundError

```java
     * 原因：
     * - 类在编译时存在，运行时被删除
     * - 静态初始化块抛出异常

    public void noClassDefFound() {

    }
}
```

### 3. 线程上下文类加载器

```java
/**
 * 线程上下文类加载器（Thread Context ClassLoader）
 *
 * 作用：
 * - 解决父加载器无法访问子加载器资源的问题
 * - SPI 机制的核心
 *
 * 默认值：
 * - 主线程：AppClassLoader
 * - 子线程：继承父线程的上下文类加载器
 *
 * Spring Boot 中：
 * - 主线程的上下文类加载器是 LaunchedURLClassLoader
 * - 确保 SPI 能正确加载 BOOT-INF/lib 中的实现类
 */
public class ContextClassLoaderDemo {

    public static void main(String[] args) {
        // 获取当前线程的上下文类加载器
        ClassLoader cl = Thread.currentThread().getContextClassLoader();
        System.out.println(cl);
        // Spring Boot 中：LaunchedURLClassLoader

        // 手动设置上下文类加载器
        ClassLoader custom = new CustomClassLoader();
        Thread.currentThread().setContextClassLoader(custom);

        try {
            // 在这里执行的代码使用 custom 作为上下文类加载器
            ServiceLoader.load(SomeInterface.class);
        } finally {
            // 恢复原来的类加载器
            Thread.currentThread().setContextClassLoader(cl);
        }
    }
}
```

### 4. Spring Boot 多模块类加载

```java
/**
 * 多模块项目中的类加载注意事项：
 *
 * 场景：主模块 + 插件模块，插件需要动态加载
 */
@Component
public class PluginLoader {

    private final Map<String, ClassLoader> pluginLoaders = new ConcurrentHashMap<>();

    public void loadPlugin(String pluginPath) throws Exception {
        URL pluginUrl = new File(pluginPath).toURI().toURL();

        // 每个插件使用独立的类加载器
        // parent 设为当前上下文类加载器，可以访问主应用的类
        URLClassLoader pluginLoader = new URLClassLoader(
            new URL[]{pluginUrl},
            Thread.currentThread().getContextClassLoader()
        );

        pluginLoaders.put(pluginPath, pluginLoader);

        // 加载插件主类
        Class<?> pluginClass = pluginLoader.loadClass("com.plugin.Main");
        Object plugin = pluginClass.getDeclaredConstructor().newInstance();

        // 调用插件方法
        pluginClass.getMethod("start").invoke(plugin);
    }

    public void unloadPlugin(String pluginPath) throws Exception {
        ClassLoader loader = pluginLoaders.remove(pluginPath);
        if (loader instanceof URLClassLoader) {
            ((URLClassLoader) loader).close();
            // 关闭后，该加载器加载的类会被 GC 回收
        }
    }
}
```

## 五、核心对比总结

```
┌──────────────────┬──────────────────────┬──────────────────────────┐
│   维度            │   JDK 标准类加载      │   Spring Boot 类加载      │
├──────────────────┼──────────────────────┼──────────────────────────┤
│ 类加载器          │ Bootstrap/Ext/App    │ + LaunchedURLClassLoader  │
│ 委派模型          │ 标准双亲委派          │ 略有调整（优先自己加载）   │
│ 加载来源          │ classpath            │ BOOT-INF/classes + lib    │
│ 嵌套 jar         │ 不支持               │ 支持（自定义 URL 协议）    │
│ 热重载            │ 不支持               │ DevTools 双类加载器方案    │
│ SPI 支持          │ 线程上下文类加载器    │ 同左，但 CL 是 Launched   │
│ 类冲突处理        │ 先加载者优先          │ 同左                      │
└──────────────────┴──────────────────────┴──────────────────────────┘

关键结论：
1. Spring Boot Fat Jar 的核心是 LaunchedURLClassLoader
2. 它解决了标准 URLClassLoader 无法加载嵌套 jar 的问题
3. DevTools 热重载通过双类加载器实现，只重载变化的类
4. 线程上下文类加载器在 Spring Boot 中默认是 LaunchedURLClassLoader
   确保 SPI 机制能正确找到 BOOT-INF/lib 中的实现
```
