# 多路复用详解

## 1. 什么是多路复用

多路复用（Multiplexing）是指用一个线程/连接同时监听和处理多个 I/O 事件，避免为每个连接创建一个线程。

```
传统模型：1个连接 → 1个线程（线程多了开销大）
多路复用：1个线程 → 监听N个连接（事件驱动）
```

## 2. I/O 多路复用（操作系统层）

### select

最早的多路复用方案，用 fd_set 位图监听文件描述符：

```c
fd_set readfds;
FD_SET(fd1, &readfds);
FD_SET(fd2, &readfds);
select(maxfd+1, &readfds, NULL, NULL, &timeout);
// 返回后遍历所有fd，判断哪个就绪
```

缺点：
- 最多监听 1024 个 fd
- 每次调用需要把 fd_set 从用户态拷贝到内核态
- 返回后需要遍历所有 fd 找到就绪的（O(n)）

### poll

改进了 select 的 1024 限制，用链表存储 fd，但本质问题没解决，仍需遍历。

### epoll（Linux，主流）

```c
// 创建 epoll 实例
int epfd = epoll_create(1);

// 注册感兴趣的事件
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &event);

// 等待事件就绪，只返回就绪的 fd
int n = epoll_wait(epfd, events, MAX_EVENTS, timeout);
for (int i = 0; i < n; i++) {
    // 直接处理 events[i]，无需遍历所有
}
```

epoll 优势：
- 无 fd 数量限制
- 用红黑树管理注册的 fd，O(log n) 增删
- 就绪列表只返回活跃 fd，O(1) 获取结果
- 支持边缘触发（ET）和水平触发（LT）

| 对比 | select | poll | epoll |
|------|--------|------|-------|
| fd 上限 | 1024 | 无限制 | 无限制 |
| 时间复杂度 | O(n) | O(n) | O(1) |
| 内核拷贝 | 每次全量 | 每次全量 | 只拷贝就绪 |
| 适用场景 | 连接少 | 连接少 | 高并发（推荐） |

## 3. HTTP/2 多路复用

HTTP/1.1 的问题：队头阻塞，一个请求没响应，后面的请求全部等待。

```
HTTP/1.1（串行）：
连接1: 请求A ──────响应A──── 请求B ──────响应B
                             （必须等A完成）

HTTP/2（多路复用）：
连接1: 请求A ──────────────响应A
       请求B ──────────响应B
       请求C ────────────────响应C
       （同一连接，并发传输，互不阻塞）
```

HTTP/2 核心概念：
- Stream（流）：一个请求/响应对，有唯一 ID
- Frame（帧）：最小传输单位，每个帧属于某个 Stream
- 同一连接可以有多个并发 Stream，帧交错传输

```
连接中的帧传输示意：
[Stream1:HEADERS][Stream2:HEADERS][Stream1:DATA][Stream3:HEADERS][Stream2:DATA]...
```

## 4. Netty 的多路复用实现

Netty 基于 Java NIO（底层用 epoll/kqueue），实现了高性能多路复用：

```java
// Netty 服务端核心结构
EventLoopGroup bossGroup = new NioEventLoopGroup(1);    // 接受连接
EventLoopGroup workerGroup = new NioEventLoopGroup();   // 处理 I/O

ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.group(bossGroup, workerGroup)
    .channel(NioServerSocketChannel.class)
    .childHandler(new ChannelInitializer<SocketChannel>() {
        @Override
        protected void initChannel(SocketChannel ch) {
            ch.pipeline().addLast(new MyHandler());
        }
    });

// bossGroup 的1个线程监听端口，接到连接后注册到 workerGroup
// workerGroup 的多个线程各自用 Selector（epoll）监听多个连接
```

线程模型：
```
BossGroup(1线程)
    ↓ accept 新连接
WorkerGroup(N线程，每个线程管理多个Channel)
    Thread1: Channel1, Channel2, Channel3 ...
    Thread2: Channel4, Channel5, Channel6 ...
    （每个线程内部用 Selector 多路复用）
```

## 5. Redis 单线程多路复用

Redis 用单线程 + I/O 多路复用处理所有客户端请求：

```
多个客户端连接
       ↓
   epoll 监听所有连接
       ↓
   有事件就绪 → 放入事件队列
       ↓
   单线程顺序处理事件队列
       ↓
   返回响应
```

为什么单线程还这么快：
- 纯内存操作，不涉及磁盘 I/O
- epoll 让单线程能高效处理大量连接
- 避免了多线程的锁竞争和上下文切换开销

## 6. Java NIO Selector（多路复用 API）

```java
// 创建 Selector
Selector selector = Selector.open();

// 注册 Channel 到 Selector
ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.configureBlocking(false); // 必须非阻塞
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

// 事件循环
while (true) {
    selector.select(); // 阻塞直到有事件就绪
    Set<SelectionKey> keys = selector.selectedKeys();

    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            // 处理新连接
            SocketChannel client = serverChannel.accept();
            client.configureBlocking(false);
            client.register(selector, SelectionKey.OP_READ);
        } else if (key.isReadable()) {
            // 处理读事件
            SocketChannel client = (SocketChannel) key.channel();
            ByteBuffer buffer = ByteBuffer.allocate(1024);
            client.read(buffer);
        }
    }
    keys.clear();
}
```

## 7. 多路复用 vs 多线程对比

| 对比项 | 多线程模型 | 多路复用模型 |
|--------|-----------|-------------|
| 并发方式 | 每连接一线程 | 单/少量线程监听多连接 |
| 内存开销 | 高（每线程约1MB栈） | 低 |
| 上下文切换 | 频繁 | 极少 |
| 编程复杂度 | 低 | 较高 |
| 适用场景 | 连接少、计算密集 | 高并发、I/O 密集 |
| 代表框架 | Tomcat（BIO模式） | Netty、Nginx、Redis |
