# TCP 与 UDP 差异详解

## 1. 核心对比

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 连接方式 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠，保证送达 | 不可靠，可能丢包 |
| 顺序 | 保证有序 | 不保证顺序 |
| 速度 | 较慢（有确认机制） | 快 |
| 头部大小 | 20-60 字节 | 8 字节 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有 | 无 |
| 适用场景 | HTTP、文件传输、邮件 | 视频直播、DNS、游戏 |

---

## 2. TCP 三次握手

```
Client          Server
  |                |
  |--- SYN ------> |   第一次：我要连接（seq=x）
  |                |
  |<-- SYN+ACK --- |   第二次：好的，我准备好了（seq=y, ack=x+1）
  |                |
  |--- ACK ------> |   第三次：收到，开始通信（ack=y+1）
  |                |
  |==== 数据传输 ==|
```

为什么是三次不是两次？两次握手服务端无法确认客户端能收到自己的消息，存在半连接风险。

---

## 3. TCP 四次挥手

```
Client          Server
  |                |
  |--- FIN ------> |   第一次：我发完了，要关闭
  |<-- ACK ------- |   第二次：收到，但我还有数据没发完
  |                |   （此时进入半关闭状态）
  |<-- FIN ------- |   第三次：我也发完了
  |--- ACK ------> |   第四次：好，正式关闭
```

为什么是四次不是三次？服务端收到 FIN 后可能还有数据要发，ACK 和 FIN 不能合并，所以比握手多一次。

`TIME_WAIT`：客户端发完最后 ACK 后等待 2MSL，防止最后的 ACK 丢失导致服务端重发 FIN 无人响应。

---

## 4. TCP 可靠性机制

- 序列号 + 确认应答：每个包有序号，收到后回 ACK，超时未收到 ACK 就重传
- 滑动窗口：批量发送多个包，不用每个包都等 ACK，提升吞吐量
- 拥塞控制：慢启动 → 拥塞避免 → 快重传 → 快恢复，防止把网络打爆

```
滑动窗口示意：
发送方可以连续发 [1,2,3,4,5]，收到 ACK 后窗口右移
[1][2][3][4][5][6][7][8]
 ↑_________↑
   窗口大小=5，可以连续发这5个
```

---

## 5. UDP 为什么还有人用

UDP 虽然不可靠，但可以在应用层自己实现部分可靠机制，同时保留低延迟优势：

- 视频直播：丢几帧没关系，延迟低更重要
- DNS 查询：包小，一来一回，不需要建连接
- 游戏：实时性优先，TCP 的重传反而会造成卡顿
- QUIC（HTTP/3）：基于 UDP 自己实现可靠传输，兼顾速度和可靠性

---

## 6. QUIC 简单说

HTTP/3 底层用 QUIC 替代 TCP，解决了 TCP 的队头阻塞问题：

```
TCP：一个包丢了，后面所有包都得等重传
QUIC：多路复用，一个流丢包不影响其他流
```
