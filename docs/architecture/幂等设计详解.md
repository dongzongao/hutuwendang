# 幂等设计详解

## 1. 什么是幂等

同一个操作执行一次和执行多次，结果完全一样。

```
f(f(x)) = f(x)
```

典型场景：用户重复点击支付、网络超时重试、MQ 消息重复消费。

---

## 2. 哪些操作天然幂等

- GET 查询：不修改数据，天然幂等
- DELETE：删除同一条记录多次，结果一样
- 带条件的 UPDATE：`UPDATE SET status=1 WHERE status=0`

哪些不幂等：
- INSERT：重复插入会产生多条记录
- UPDATE 累加：`UPDATE SET amount = amount + 100`，每次都会变

---

## 3. 常见幂等实现方案

### 方案一：唯一索引（数据库层）

思路：在业务唯一字段上加唯一索引，重复插入直接报 `DuplicateKeyException`，捕获异常即可。

```sql
ALTER TABLE orders ADD UNIQUE KEY uk_order_no (order_no);
```

```java
try {
    orderMapper.insert(order);
} catch (DuplicateKeyException e) {
    // 重复请求，直接返回已有结果
    return orderMapper.selectByOrderNo(order.getOrderNo());
}
```

适用：INSERT 场景，简单可靠。

---

### 方案二：Token 机制（防重提交）

思路：请求前先获取一个 token 存入 Redis，提交时携带 token，服务端用 `del` 删除并判断是否删除成功，成功才处理业务。

```
1. 客户端 GET /token  →  服务端生成 token 存 Redis，返回给客户端
2. 客户端提交请求携带 token
3. 服务端 Redis DEL token
   - 删除成功(返回1) → 执行业务
   - 删除失败(返回0) → 重复请求，直接返回
```

```java
// 关键：DEL 操作是原子的，天然防并发
Long result = redisTemplate.delete(token) ? 1L : 0L;
if (result == 0) {
    return Result.fail("重复请求");
}
// 执行业务逻辑
```

注意：获取 token 和删除 token 必须是原子操作，可用 Lua 脚本保证。

---

### 方案三：状态机（状态流转控制）

思路：业务状态只能单向流转，用 `WHERE status = 期望状态` 做乐观锁，更新失败说明已被处理。

```sql
-- 只有 status=待支付 才能更新为已支付
UPDATE orders SET status = 'PAID' WHERE order_id = ? AND status = 'PENDING'
```

```java
int rows = orderMapper.updateStatus(orderId, "PENDING", "PAID");
if (rows == 0) {
    // 状态已变更，说明已处理过
    return;
}
```

适用：订单状态流转、审批流等场景。

---

### 方案四：Redis SETNX（分布式去重）

思路：用请求唯一标识（如 requestId）做 key，SETNX 设置成功才处理，失败说明已在处理或已处理过。

```java
String key = "idempotent:" + requestId;
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent(key, "1", 10, TimeUnit.MINUTES);

if (!success) {
    return Result.fail("重复请求");
}
// 执行业务逻辑
```

注意：key 的过期时间要根据业务合理设置，防止 key 永久占用。

---

### 方案五：MQ 消息去重

思路：消费者维护一张已消费消息表，消费前查是否已处理，处理完插入记录，利用唯一索引防重。

```java
// 消费消息时
String msgId = message.getMsgId();
if (consumeLogMapper.exists(msgId)) {
    // 已消费，直接 ACK
    return;
}
// 执行业务 + 插入消费记录（同一事务）
businessService.process(message);
consumeLogMapper.insert(msgId);
```

---

## 4. 方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 唯一索引 | INSERT 去重 | 简单，DB 保证 | 依赖数据库 |
| Token 机制 | 表单防重提交 | 前后端配合，体验好 | 需要额外获取 token 接口 |
| 状态机 | 订单/审批流转 | 业务语义清晰 | 只适合有状态的业务 |
| Redis SETNX | 通用接口去重 | 性能好，分布式友好 | 需处理 Redis 故障降级 |
| MQ 消息去重 | 消息消费 | 可靠，可追溯 | 需维护消费记录表 |

---

## 5. 设计幂等的核心原则

1. 唯一标识：每个请求必须有全局唯一 ID（requestId / orderNo / msgId）
2. 原子操作：查询+处理+标记必须保证原子性，防止并发漏洞
3. 合理过期：去重标记不能永久保存，根据业务设置合理 TTL
4. 返回一致：重复请求应返回和第一次相同的结果，而不是报错
