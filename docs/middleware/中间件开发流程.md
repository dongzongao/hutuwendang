# 中间件开发流程

## 1. 什么是中间件

中间件是位于操作系统与应用程序之间的软件层，提供通用能力（通信、存储、协调、限流等），让业务代码专注于业务逻辑。

常见分类：

| 类型 | 代表产品 | 核心能力 |
|------|---------|---------|
| 消息队列 | Kafka、RocketMQ | 异步解耦、削峰填谷 |
| 注册中心 | Nacos、ZooKeeper | 服务发现、配置管理 |
| RPC 框架 | Dubbo、gRPC | 远程调用、负载均衡 |
| 网关 | Gateway、Nginx | 路由、限流、鉴权 |
| 缓存 | Redis、Memcached | 高速读写、数据缓冲 |
| 数据库连接池 | HikariCP、Druid | 连接复用、监控 |

---

## 2. 整体开发流程

```
需求分析 → 架构设计 → 核心模块实现 → SPI 扩展 → 测试 → 部署上线
```

### 2.1 需求分析

明确以下核心问题：

- **功能边界**：这个中间件解决什么问题，不解决什么问题
- **性能目标**：QPS、延迟 P99、吞吐量指标
- **可靠性要求**：是否需要持久化、是否允许消息丢失
- **扩展性**：是否需要水平扩展、是否需要插件机制

### 2.2 架构设计

核心设计决策：

```
1. 通信模型：同步/异步，长连接/短连接
2. 线程模型：单线程/多线程/Reactor
3. 存储模型：内存/磁盘/混合
4. 协议设计：自定义二进制协议 or 基于现有协议
5. 高可用方案：主从/集群/分片
```

## 3. 核心模块实现

### 3.1 网络通信层（Netty）

绝大多数 Java 中间件基于 Netty 构建网络层。

```java
// 服务端启动模板
ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.group(bossGroup, workerGroup)
    .channel(NioServerSocketChannel.class)
    .option(ChannelOption.SO_BACKLOG, 1024)
    .childOption(ChannelOption.TCP_NODELAY, true)
    .childHandler(new ChannelInitializer<SocketChannel>() {
        @Override
        protected void initChannel(SocketChannel ch) {
            ch.pipeline()
                .addLast(new LengthFieldBasedFrameDecoder(65536, 0, 4, 0, 4))
                .addLast(new ProtocolDecoder())
                .addLast(new ProtocolEncoder())
                .addLast(new BusinessHandler());
        }
    });
```

关键参数：

| 参数 | 说明 | 推荐值 |
|------|------|-------|
| `SO_BACKLOG` | 连接队列长度 | 1024 |
| `TCP_NODELAY` | 禁用 Nagle 算法 | true（低延迟场景） |
| `SO_KEEPALIVE` | TCP 保活 | true |
| `SO_RCVBUF/SO_SNDBUF` | 收发缓冲区 | 根据吞吐量调整 |

### 3.2 协议设计

自定义二进制协议通常包含：

```
+--------+--------+--------+--------+--------+--------+
| magic  | version| type   | status | req_id | length |
| 2 byte | 1 byte | 1 byte | 1 byte | 8 byte | 4 byte |
+--------+--------+--------+--------+--------+--------+
|                    body (length bytes)               |
+------------------------------------------------------+
```

- `magic`：魔数，用于快速识别协议（如 `0xCAFE`）
- `version`：协议版本，支持协议升级
- `type`：请求类型（REQUEST/RESPONSE/HEARTBEAT）
- `req_id`：请求 ID，用于异步响应匹配
- `length`：body 长度，防止粘包

### 3.3 编解码

```java
// 解码器：基于 LengthFieldBasedFrameDecoder 处理粘包
public class ProtocolDecoder extends MessageToMessageDecoder<ByteBuf> {
    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        short magic = in.readShort();
        if (magic != MAGIC) {
            ctx.close();
            return;
        }
        byte version = in.readByte();
        byte type = in.readByte();
        long reqId = in.readLong();
        int bodyLen = in.readInt();
        byte[] body = new byte[bodyLen];
        in.readBytes(body);
        out.add(new RemotingCommand(type, reqId, body));
    }
}
```

### 3.4 线程模型

Reactor 三种模式：

```
单 Reactor 单线程：所有 IO + 业务在一个线程，适合简单场景
单 Reactor 多线程：IO 单线程，业务线程池处理，Netty 默认模式
主从 Reactor 多线程：bossGroup 接受连接，workerGroup 处理 IO，业务线程池处理逻辑
```

```java
// 主从 Reactor 配置
EventLoopGroup bossGroup = new NioEventLoopGroup(1);       // 接受连接
EventLoopGroup workerGroup = new NioEventLoopGroup(8);     // 处理 IO
ExecutorService businessPool = new ThreadPoolExecutor(     // 处理业务
    16, 64, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>(10000)
);
```

业务 Handler 中将耗时操作提交到业务线程池，避免阻塞 IO 线程：

```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    businessPool.submit(() -> {
        Object result = process(msg);
        ctx.writeAndFlush(result);
    });
}
```

### 3.5 连接管理

```java
// 连接池：复用长连接，避免频繁建连
public class ConnectionPool {
    private final ConcurrentHashMap<String, Channel> pool = new ConcurrentHashMap<>();

    public Channel getOrCreate(String address) {
        return pool.computeIfAbsent(address, addr -> {
            Bootstrap bootstrap = new Bootstrap();
            // ... 建立连接
            return bootstrap.connect(host, port).sync().channel();
        });
    }

    // 心跳检测：定时发送 HEARTBEAT，超时关闭连接
    public void startHeartbeat(Channel channel) {
        channel.pipeline().addLast(new IdleStateHandler(0, 0, 30, TimeUnit.SECONDS));
        channel.pipeline().addLast(new HeartbeatHandler());
    }
}
```

## 4. SPI 扩展机制

中间件通常通过 SPI（Service Provider Interface）提供扩展点，让用户自定义实现。

### 4.1 Java 原生 SPI

```
// 1. 定义接口
public interface Serializer {
    byte[] serialize(Object obj);
    <T> T deserialize(byte[] bytes, Class<T> clazz);
}

// 2. 在 META-INF/services/ 下创建文件
// 文件名：com.example.Serializer
// 内容：
com.example.JsonSerializer
com.example.ProtobufSerializer

// 3. 加载
ServiceLoader<Serializer> loader = ServiceLoader.load(Serializer.class);
```

### 4.2 增强版 SPI（Dubbo 风格）

原生 SPI 的问题：一次性加载所有实现，无法按需加载、无法注入依赖。

Dubbo ExtensionLoader 解决方案：

```java
// 接口标注 @SPI 指定默认实现
@SPI("json")
public interface Serializer {
    byte[] serialize(Object obj);
}

// 配置文件 META-INF/extensions/com.example.Serializer
json=com.example.JsonSerializer
protobuf=com.example.ProtobufSerializer

// 按名称加载，懒加载
Serializer serializer = ExtensionLoader
    .getExtensionLoader(Serializer.class)
    .getExtension("protobuf");
```

核心特性：
- 懒加载：只实例化用到的扩展
- 自动注入：扩展实现中的 setter 自动注入其他扩展
- AOP 包装：Wrapper 类自动包裹扩展，实现切面逻辑
- 自适应扩展：运行时根据 URL 参数动态选择实现

## 5. 序列化选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| JSON（Jackson） | 可读性好，生态成熟 | 体积大，性能一般 | 调试、对外接口 |
| Protobuf | 体积小，性能高，跨语言 | 需要 .proto 文件，可读性差 | 高性能 RPC |
| Kryo | Java 专用，性能极高 | 不跨语言，版本兼容问题 | 内部 Java 系统 |
| Hessian | 跨语言，无需 schema | 性能中等 | Dubbo 默认 |

性能参考（序列化 1000 次 1KB 对象）：

```
Protobuf:  ~0.5ms
Kryo:      ~0.8ms
Hessian:   ~2ms
JSON:      ~3ms
Java 原生: ~10ms
```

## 6. 测试策略

### 6.1 单元测试

针对编解码、序列化、路由算法等纯逻辑模块：

```java
@Test
void testEncodeDecode() {
    RemotingCommand cmd = new RemotingCommand(REQUEST, 1L, "hello".getBytes());
    ByteBuf buf = encoder.encode(cmd);
    RemotingCommand decoded = decoder.decode(buf);
    assertEquals(cmd.getReqId(), decoded.getReqId());
    assertArrayEquals(cmd.getBody(), decoded.getBody());
}
```

### 6.2 集成测试

启动真实 Server，验证完整请求链路：

```java
@Test
void testRequestResponse() throws Exception {
    Server server = new Server(8080);
    server.start();
    Client client = new Client("localhost", 8080);
    String result = client.send("ping");
    assertEquals("pong", result);
    server.shutdown();
}
```

### 6.3 性能测试

使用 JMH 进行基准测试：

```java
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.SECONDS)
@State(Scope.Benchmark)
public class SerializerBenchmark {
    private JsonSerializer json = new JsonSerializer();
    private ProtobufSerializer proto = new ProtobufSerializer();
    private User user = new User("test", 18);

    @Benchmark
    public byte[] jsonSerialize() { return json.serialize(user); }

    @Benchmark
    public byte[] protoSerialize() { return proto.serialize(user); }
}
```

压测关注指标：
- **吞吐量**：TPS/QPS
- **延迟分布**：P50/P95/P99
- **资源消耗**：CPU、内存、GC 频率
- **连接数**：最大并发连接数

## 7. 生产部署要点

### 7.1 优雅启动与关闭

```java
// 注册 JVM 关闭钩子
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    // 1. 停止接受新请求
    server.pauseAccept();
    // 2. 等待处理中的请求完成（最多 30s）
    awaitTermination(30, TimeUnit.SECONDS);
    // 3. 关闭连接池、线程池
    connectionPool.shutdown();
    businessPool.shutdown();
    // 4. 释放端口
    server.close();
}));
```

### 7.2 监控指标暴露

通过 Micrometer 暴露关键指标：

```java
// 注册指标
Metrics.gauge("middleware.connections.active", connectionPool, ConnectionPool::activeCount);
Metrics.counter("middleware.requests.total").increment();
Timer.Sample sample = Timer.start();
// ... 处理请求
sample.stop(Metrics.timer("middleware.request.duration"));
```

关键监控项：

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| 活跃连接数 | 当前建立的连接数 | > 最大连接数 80% |
| 请求队列深度 | 等待处理的请求数 | > 队列容量 70% |
| 请求耗时 P99 | 99 分位延迟 | 超过 SLA 要求 |
| GC 停顿时间 | Full GC 频率与耗时 | Full GC > 1次/分钟 |
| 错误率 | 请求失败比例 | > 0.1% |

### 7.3 配置热更新

```java
// 监听配置变更，动态调整线程池参数
configCenter.addListener("middleware.thread.pool.size", newValue -> {
    int newSize = Integer.parseInt(newValue);
    ((ThreadPoolExecutor) businessPool).setCorePoolSize(newSize);
    ((ThreadPoolExecutor) businessPool).setMaximumPoolSize(newSize * 2);
    log.info("Thread pool size updated to {}", newSize);
});
```

## 8. 常见问题与解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 粘包/拆包 | TCP 流式传输 | LengthFieldBasedFrameDecoder 定长帧 |
| 连接泄漏 | 异常未关闭 Channel | try-with-resources + IdleStateHandler 超时关闭 |
| 线程池打满 | 业务处理慢 | 限流 + 快速失败 + 扩容 |
| 序列化兼容 | 字段增删 | 使用 Protobuf/Hessian，避免 Java 原生序列化 |
| 脑裂 | 网络分区 | Raft/ZAB 共识算法，奇数节点部署 |
