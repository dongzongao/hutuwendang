# BM25 算法详解

## 是什么

BM25（Best Match 25）是 ES 默认的相关性评分算法，用于计算文档与查询词的匹配程度，决定搜索结果的排序。

ES 5.0 之前使用 TF-IDF，5.0 之后默认切换为 BM25。

---

## 公式

```
Score(D, Q) = Σ IDF(qi) × TF_norm(qi, D)

其中：

IDF(qi) = ln(1 + (N - n(qi) + 0.5) / (n(qi) + 0.5))
  N      = 文档总数
  n(qi)  = 包含词 qi 的文档数

TF_norm(qi, D) = f(qi,D) × (k1 + 1) / (f(qi,D) + k1 × (1 - b + b × |D|/avgdl))
  f(qi,D) = 词 qi 在文档 D 中的出现次数
  |D|     = 文档长度（词数）
  avgdl   = 所有文档的平均长度
  k1      = 词频饱和参数，默认 1.2
  b       = 文档长度归一化参数，默认 0.75
```

---

## 核心思想

### IDF（逆文档频率）

词越稀有，权重越高。

```
"的" 出现在 99% 的文档中 → IDF 接近 0，几乎不贡献分数
"Elasticsearch" 只出现在 1% 的文档中 → IDF 高，贡献大
```

### TF（词频）饱和

BM25 对词频做了饱和处理，词频越高分数增长越慢，避免通过堆砌关键词刷分：

```
TF-IDF：词频线性增长，出现 100 次比 10 次分数高 10 倍
BM25：  词频饱和，出现 100 次和 10 次差距很小
```

```
词频 vs 分数（k1=1.2）：
词频=1  → 分数≈0.52
词频=5  → 分数≈0.83
词频=10 → 分数≈0.91
词频=100→ 分数≈0.99  ← 趋近上限
```

### 文档长度归一化

短文档中出现关键词比长文档更有意义：

```
文档A（10词）：包含 "Java" 1次  → 相关性高
文档B（1000词）：包含 "Java" 1次 → 相关性低

b=0.75 控制长度归一化的强度：
  b=0 → 完全不考虑文档长度
  b=1 → 完全按文档长度归一化
```

---

## BM25 vs TF-IDF 对比

| 对比项 | TF-IDF | BM25 |
|--------|--------|------|
| 词频处理 | 线性增长 | 饱和上限 |
| 文档长度 | 简单归一化 | 可调参数控制 |
| 参数调节 | 无 | k1、b 可调 |
| 抗刷词 | 弱 | 强 |
| 实际效果 | 一般 | 更好 |

## 在 ES 中调整参数

```json
PUT /my_index
{
  "settings": {
    "similarity": {
      "custom_bm25": {
        "type": "BM25",
        "k1": 1.5,
        "b": 0.8
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "similarity": "custom_bm25"
      }
    }
  }
}
```

- 增大 `k1`：词频影响更大，适合长文档搜索
- 减小 `b`：降低文档长度的影响，适合文档长度差异大的场景

## boost 手动调权

```json
{
  "multi_match": {
    "query": "Java 并发",
    "fields": ["title^3", "content^1"]
  }
}
```

`title^3` 表示 title 字段的分数乘以 3，让标题匹配的文档排名更靠前。
