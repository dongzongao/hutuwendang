# 红黑树详解

## 是什么

红黑树是一种自平衡二叉搜索树，通过对节点染色（红/黑）和旋转操作，保证树的高度始终在 O(log n) 级别，从而保证查找、插入、删除都是 O(log n)。

---

## 五条性质

1. 每个节点是红色或黑色
2. 根节点是黑色
3. 叶子节点（NIL 节点）是黑色
4. 红色节点的两个子节点必须是黑色（不能有连续红节点）
5. 从任意节点到其所有叶子节点的路径上，黑色节点数量相同（黑高相等）

这五条性质共同保证：最长路径不超过最短路径的 2 倍，树高 ≤ 2log(n+1)。

---

## 基本操作

### 左旋

```
    X                Y
   / \              / \
  A   Y    →      X   C
     / \         / \
    B   C       A   B
```

### 右旋

```
      Y            X
     / \          / \
    X   C   →   A   Y
   / \              / \
  A   B            B   C
```

旋转不改变中序遍历顺序，只调整结构。

---

## 插入

新节点默认染红色（不影响黑高），然后根据叔父节点颜色修复：

```
情况1：叔父节点是红色
  → 父节点和叔父节点变黑，祖父节点变红，向上递归修复

情况2：叔父节点是黑色，新节点是父节点的内侧子节点
  → 先对父节点旋转，转化为情况3

情况3：叔父节点是黑色，新节点是父节点的外侧子节点
  → 对祖父节点旋转，交换父节点和祖父节点颜色
```

示例（插入节点 4）：

```
插入前：
      5(黑)
     / \
   3(红) 7(红)

插入 4（红）：
      5(黑)
     / \
   3(红) 7(红)
    \
    4(红)  ← 连续红节点，违反性质4

叔父 7 是红色 → 情况1：3和7变黑，5变红，5是根再变黑

      5(黑)
     / \
   3(黑) 7(黑)
    \
    4(红)  ✅
```

---

## 删除

删除比插入复杂，核心是处理"双重黑"节点（删除黑节点后黑高不平衡）：

```
情况1：兄弟节点是红色 → 旋转，转化为情况2/3/4
情况2：兄弟节点是黑色，兄弟的两个子节点都是黑色 → 兄弟变红，向上传递
情况3：兄弟节点是黑色，兄弟的近侧子节点是红色 → 旋转兄弟，转化为情况4
情况4：兄弟节点是黑色，兄弟的远侧子节点是红色 → 旋转父节点，调整颜色，结束
```

## 与其他树的对比

| 对比项 | BST | AVL 树 | 红黑树 |
|--------|-----|--------|--------|
| 查找 | O(n) 最坏 | O(log n) | O(log n) |
| 插入 | O(n) 最坏 | O(log n) | O(log n) |
| 删除 | O(n) 最坏 | O(log n) | O(log n) |
| 平衡方式 | 无 | 严格平衡（高度差≤1） | 近似平衡（黑高相等） |
| 旋转次数（插入） | 0 | ≤2 | ≤2 |
| 旋转次数（删除） | 0 | O(log n) | ≤3 |
| 适用场景 | 数据有序 | 读多写少 | 读写均衡 |

AVL 树比红黑树更严格平衡，查找略快，但插入删除需要更多旋转。红黑树在写操作频繁时性能更好。

---

## 实际应用

| 场景 | 说明 |
|------|------|
| Java `TreeMap` / `TreeSet` | 基于红黑树，保证 key 有序，O(log n) 操作 |
| Java `HashMap` | 链表长度 > 8 时转为红黑树，防止哈希冲突退化 |
| Java `ConcurrentHashMap` | 同上 |
| Linux 内核 CFS 调度器 | 用红黑树管理进程调度队列 |
| Nginx 定时器 | 用红黑树管理超时事件 |

---

## 与 B+ 树的区别

| 对比项 | 红黑树 | B+ 树 |
|--------|--------|-------|
| 节点子节点数 | 2（二叉） | M 路（多叉） |
| 树高 | O(log n) | O(log_M n)，更矮 |
| 磁盘 IO | 多（树高较高） | 少（专为磁盘设计） |
| 适用场景 | 内存数据结构 | 数据库索引、文件系统 |

红黑树适合内存中的有序数据，B+ 树适合磁盘存储，两者各有专长。
