# B 树与 B+ 树详解

## B 树（B-Tree）

每个节点既存 key 也存 data，叶子节点之间没有链接。

```
        [30]
       /    \
   [10,20]  [40,50]
   /  |  \   /  |  \
 [d] [d] [d][d] [d] [d]
```

## B+ 树（B+Tree）

内部节点只存 key 做索引，data 全部存在叶子节点，叶子节点之间用链表串联。

```
         [30]
        /    \
   [10,20]  [30,40,50]
      ↓           ↓
[10→20→30→40→50]  ← 叶子链表
```

---

## 核心区别对比

| 对比项 | B 树 | B+ 树 |
|--------|------|-------|
| 数据存储位置 | 所有节点都存 data | 只有叶子节点存 data |
| 叶子节点链表 | 无 | 有，支持顺序遍历 |
| 单次查询 | 最好情况在根节点就命中 | 必须走到叶子节点 |
| 范围查询 | 需要中序遍历整棵树 | 直接在叶子链表上扫描，效率高 |
| 同等阶数，节点容量 | 较小（节点存了 data） | 更大（内部节点只存 key） |
| 树高 | 相对较高 | 相对更矮，IO 次数更少 |

---

## MySQL 为什么选 B+ 树

1. 内部节点不存 data，单个节点能放更多 key，树更矮，磁盘 IO 更少
2. 叶子链表天然支持范围查询（`BETWEEN`、`ORDER BY`、`LIMIT`）
3. 查询路径稳定，每次都走到叶子，性能可预测

B 树的优势在于单点查询可能更快（中间节点命中就返回），所以在一些 key-value 存储场景有应用，但对数据库的范围查询场景不友好。
