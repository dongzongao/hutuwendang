# 幻读详解

## 什么是幻读

同一事务内，两次执行相同的范围查询，第二次查询返回了第一次没有的行，就像出现了"幻影"。

```sql
-- 事务A
BEGIN;
SELECT * FROM user WHERE age > 18;  -- 返回 3 条

-- 事务B（此时提交）
INSERT INTO user (name, age) VALUES ('Tom', 20);
COMMIT;

-- 事务A 再次查询
SELECT * FROM user WHERE age > 18;  -- 返回 4 条 ← 幻读
```

---

## 幻读 vs 不可重复读

| 现象 | 描述 | 操作类型 |
|------|------|----------|
| 不可重复读 | 同一行数据两次读取结果不同 | UPDATE / DELETE |
| 幻读 | 两次范围查询行数不同 | INSERT |

---

## MVCC 解决了快照读的幻读

在 `REPEATABLE READ` 隔离级别下，普通 `SELECT` 是快照读，读的是事务开始时的 Read View 版本，其他事务插入的新行对当前事务不可见。

```sql
BEGIN;
SELECT * FROM user WHERE age > 18;  -- 生成 Read View，返回 3 条

-- 事务B 插入并提交

SELECT * FROM user WHERE age > 18;  -- 仍读旧版本，返回 3 条 ✅
COMMIT;
```

MVCC 通过版本链让快照读不受其他事务插入影响，解决了快照读场景下的幻读。

---

## 间隙锁解决当前读的幻读

当前读（`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`）会读最新数据并加锁，MVCC 无法解决这种场景的幻读，需要间隙锁（Gap Lock）。

```sql
BEGIN;
-- 当前读，加 Next-Key Lock，锁住 (18, +∞) 的间隙
SELECT * FROM user WHERE age > 18 FOR UPDATE;

-- 事务B 尝试插入 age=20 的行
INSERT INTO user (name, age) VALUES ('Tom', 20);
-- 被间隙锁阻塞，等待事务A 提交

-- 事务A 再次当前读
SELECT * FROM user WHERE age > 18 FOR UPDATE;  -- 仍是 3 条 ✅
COMMIT;
-- 事务B 的插入此时才能执行
```

---

## 加锁范围分析

以主键索引为例，数据：`id = 1, 5, 10, 15, 20`

```sql
-- 范围查询加锁
SELECT * FROM t WHERE id > 8 FOR UPDATE;

加锁范围（Next-Key Lock）：
(5, 10]  (10, 15]  (15, 20]  (20, +∞)

其他事务无法在这些区间内插入任何数据
```

---

## RR 下幻读是否被彻底解决

**结论：不完全是。**

存在一种特殊场景，在同一事务内混用快照读和当前读，会出现幻读：

```sql
BEGIN;

-- 1. 快照读，生成 Read View，看不到后续插入
SELECT * FROM user WHERE id = 5;  -- 返回空

-- 事务B 插入 id=5 并提交

-- 2. 当前读，读最新数据
SELECT * FROM user WHERE id = 5 FOR UPDATE;  -- 返回 1 条 ← 幻读！

-- 3. 或者直接 UPDATE
UPDATE user SET name = 'x' WHERE id = 5;  -- 影响 1 行
SELECT * FROM user WHERE id = 5;          -- 返回 1 条 ← 幻读！
COMMIT;
```

原因：`UPDATE` 是当前读，修改了事务B 插入的行后，该行的 `trx_id` 变成了当前事务，后续快照读可以看到它。

---

## 总结

| 场景 | 解决方案 | 是否彻底 |
|------|----------|----------|
| 快照读（普通 SELECT） | MVCC | ✅ 彻底解决 |
| 当前读（FOR UPDATE / UPDATE） | Next-Key Lock（间隙锁） | ✅ 彻底解决 |
| 快照读与当前读混用 | 无法完全避免 | ❌ 存在边界场景 |

实际开发中，在同一事务内避免混用快照读和当前读，可以规避幻读问题。
