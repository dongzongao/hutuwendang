# 开发方法论

## 1. 需求分析

拿到需求不要急着写代码，先搞清楚这几个问题：

- 这个功能解决什么问题？
- 边界条件是什么？异常场景有哪些？
- 数据量级是多少？需要考虑性能吗？
- 和现有系统如何交互？

**好的习惯：** 用自己的话把需求复述一遍，和产品/需求方确认，避免理解偏差导致返工。

---

## 2. 技术方案设计

### 先设计再编码

```
需求 → 方案设计 → 评审 → 编码 → 测试 → 上线
```

不要跳过设计直接写代码，尤其是涉及多人协作或复杂业务的场景。

### 方案设计要考虑的维度

- 功能正确性：核心流程是否覆盖
- 性能：接口响应时间、数据库查询效率
- 可扩展性：未来需求变化时改动成本
- 可维护性：代码是否易读、易改
- 安全性：权限控制、数据校验、SQL 注入等
- 容错性：下游服务挂了怎么办？数据库超时怎么处理？

## 3. 编码原则

### SOLID 原则

| 原则 | 说明 |
|------|------|
| S - 单一职责 | 一个类只做一件事 |
| O - 开闭原则 | 对扩展开放，对修改关闭 |
| L - 里氏替换 | 子类可以替换父类 |
| I - 接口隔离 | 接口粒度要小，不强迫实现不需要的方法 |
| D - 依赖倒置 | 依赖抽象，不依赖具体实现 |

### 命名规范

```java
// 类名：大驼峰，名词
class OrderService {}

// 方法名：小驼峰，动词开头
void createOrder() {}
boolean isValid() {}
List<Order> getOrderList() {}

// 变量名：小驼峰，见名知意
int orderCount = 0;       // ✓
int n = 0;                // ✗ 含义不明

// 常量：全大写下划线分隔
static final int MAX_RETRY_COUNT = 3;

// 布尔变量：is/has/can 开头
boolean isActive;
boolean hasPermission;
```

### 方法设计

```java
// 方法不超过 20 行，超过就考虑拆分
// 参数不超过 3 个，超过用对象封装
// 一个方法只做一件事

// ✗ 一个方法做太多事
public void processOrder(Order order) {
    // 校验参数
    // 查询库存
    // 扣减库存
    // 创建订单
    // 发送通知
    // 记录日志
}

// ✓ 拆分职责
public void processOrder(Order order) {
    validateOrder(order);
    inventoryService.deduct(order);
    orderRepository.save(order);
    notificationService.send(order);
}
```

## 4. 代码审查（Code Review）

### 提交 PR 前自查

- [ ] 代码是否有明显的 bug？
- [ ] 是否有未处理的异常？
- [ ] 是否有硬编码的魔法值？
- [ ] 日志是否合理（不要打敏感信息）？
- [ ] 是否有内存泄漏风险（ThreadLocal、流未关闭等）？
- [ ] 并发场景是否考虑线程安全？
- [ ] 数据库操作是否有索引支持？

### Review 别人代码时

- 关注逻辑正确性，不要只看格式
- 提建议而不是命令，用"建议"、"可以考虑"
- 区分必须改和可以改，不要让 PR 卡在风格问题上

## 5. 测试策略

### 测试金字塔

```
        /\
       /  \
      / E2E \        少量，成本高
     /--------\
    / 集成测试  \     适量
   /------------\
  /   单元测试   \   大量，成本低
 /--------------\
```

### 单元测试原则

```java
// 一个测试只验证一件事
// 测试方法命名：方法名_场景_期望结果
@Test
void createOrder_whenStockInsufficient_shouldThrowException() {
    // Given
    Order order = new Order(productId, quantity: 100);
    when(inventoryService.getStock(productId)).thenReturn(50);

    // When & Then
    assertThrows(InsufficientStockException.class,
        () -> orderService.createOrder(order));
}
```

## 6. 性能优化思路

### 先测量再优化

```
发现问题 → 定位瓶颈 → 制定方案 → 实施 → 验证效果
```

不要凭感觉优化，先用工具定位瓶颈：
- 接口慢：APM 工具（SkyWalking、Arthas）
- SQL 慢：慢查询日志、EXPLAIN
- 内存问题：JVM heap dump 分析
- CPU 高：线程 dump、火焰图

### 常见优化手段

```
数据库层：
  - 加索引（最常见）
  - 避免 SELECT *
  - 批量操作代替循环单条
  - 读写分离

缓存层：
  - 热点数据加 Redis 缓存
  - 本地缓存（Caffeine）减少 Redis 访问
  - 注意缓存穿透、击穿、雪崩

代码层：
  - 减少不必要的对象创建
  - 集合初始化指定容量
  - 异步处理非核心逻辑
  - 并行处理独立任务
```

## 7. 上线流程

### 发布前检查

- [ ] 数据库变更脚本是否已执行（或准备好）？
- [ ] 配置项是否已更新？
- [ ] 是否有不兼容的接口变更？
- [ ] 回滚方案是否准备好？
- [ ] 灰度发布还是全量发布？

### 灰度发布策略

```
全量发布风险高时，优先灰度：

1% 流量 → 观察指标 → 5% → 观察 → 20% → 观察 → 100%

观察指标：错误率、响应时间、业务成功率
```

### 上线后监控

- 错误日志是否有异常增加？
- 接口响应时间是否正常？
- 数据库连接数、慢查询是否正常？
- 业务指标（订单量、支付成功率等）是否正常？

## 8. 故障处理

### 故障响应优先级

```
1. 止血（先恢复服务，再查原因）
   - 回滚代码
   - 切换流量
   - 降级/熔断

2. 定位（找到根因）
   - 查日志
   - 看监控
   - 复现问题

3. 修复（彻底解决）
   - 修复代码
   - 补充测试
   - 完善监控告警

4. 复盘（避免再次发生）
   - 写故障报告
   - 总结根因
   - 制定改进措施
```

### 故障报告模板

```
时间：
影响范围：
根本原因：
处理过程：
恢复时间：
改进措施：
```

## 9. 技术债务管理

技术债务不可避免，关键是要管理好：

- 记录：发现技术债务时在代码里加 `// TODO:` 或在任务系统里记录
- 评估：定期评估债务的影响和修复成本
- 还债：每个迭代留出 10-20% 时间处理技术债务
- 不要让债务积累到影响正常开发

## 10. 持续学习

技术更新快，保持学习习惯：

- 关注官方文档和 Release Notes，而不只是博客
- 遇到问题先看源码，不要只靠搜索
- 做完一个功能后复盘：有没有更好的方案？
- 定期回顾自己写的旧代码，看看有什么可以改进的
