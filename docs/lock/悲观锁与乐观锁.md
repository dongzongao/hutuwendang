# 悲观锁与乐观锁

## 悲观锁

假设并发冲突一定会发生，操作前先加锁，其他事务阻塞等待。

```sql
BEGIN;
SELECT * FROM order WHERE id = 1 FOR UPDATE;  -- 加排他锁
UPDATE order SET status = 'paid' WHERE id = 1;
COMMIT;
```

- 依赖数据库锁机制，锁持有期间其他事务无法修改
- 适合写多读少、冲突概率高的场景
- 高并发下锁竞争严重，吞吐量低

---

## 乐观锁

假设并发冲突很少发生，不加锁，提交时检查数据是否被修改，冲突则重试。

### 版本号方式

```sql
-- 表加 version 字段
SELECT id, status, version FROM order WHERE id = 1;
-- 返回 version = 3

-- 更新时校验 version
UPDATE order SET status = 'paid', version = version + 1
WHERE id = 1 AND version = 3;

-- affected rows = 1 → 成功
-- affected rows = 0 → 冲突，重试
```

### 时间戳方式

```sql
UPDATE order SET status = 'paid', update_time = NOW()
WHERE id = 1 AND update_time = '2024-01-01 10:00:00';
```

---

## 核心对比

| 对比项 | 悲观锁 | 乐观锁 |
|--------|--------|--------|
| 加锁时机 | 操作前加锁 | 提交时校验 |
| 并发性能 | 低（锁等待） | 高（无锁等待） |
| 冲突处理 | 阻塞等待 | 检测冲突后重试 |
| 适用场景 | 写多、冲突概率高 | 读多写少、冲突概率低 |
| 实现方式 | `FOR UPDATE` | version 字段、CAS |
| 死锁风险 | 有 | 无 |

---

## 乐观锁的问题

### ABA 问题

```
线程A 读取 version=1
线程B 修改 1→2，再改回 2→1
线程A 提交时 version 仍是 1，误以为未被修改
```

解决：使用单调递增版本号（不允许回退）。

### 重试风暴

高并发下大量事务同时失败重试，加重数据库压力。
解决：加随机退避（random backoff）或限制最大重试次数。

## Java 中的乐观锁（CAS）

```java
AtomicInteger count = new AtomicInteger(0);

// CAS：期望值是 0，更新为 1
boolean success = count.compareAndSet(0, 1);

// 底层是 CPU 的 cmpxchg 指令，原子操作，无需加锁
```

`java.util.concurrent.atomic` 包下所有原子类底层都是 CAS，本质是乐观锁。
