# ZooKeeper 分布式锁详解

## 核心机制

ZooKeeper 分布式锁基于两个特性：

1. 临时节点：客户端断开连接后节点自动删除，天然解决死锁问题
2. 顺序节点：创建时自动追加递增序号，保证公平性

---

## 加锁流程

```
1. 在 /locks/my_lock/ 下创建临时顺序节点
   → /locks/my_lock/lock-0000000001

2. 获取 /locks/my_lock/ 下所有子节点，排序

3. 判断自己是否是序号最小的节点
   是 → 加锁成功
   否 → 监听（Watch）前一个节点的删除事件，等待

4. 前一个节点删除 → 收到通知 → 重新判断是否最小
```

```
节点列表：
lock-0000000001  ← 持有锁
lock-0000000002  ← 监听 001
lock-0000000003  ← 监听 002

001 释放锁（删除节点）→ 002 收到通知 → 002 获得锁
```

---

## 为什么监听前一个节点而不是最小节点

如果所有节点都监听最小节点，最小节点删除时会同时通知所有等待节点，造成惊群效应（Herd Effect）。

监听前一个节点，每次只唤醒一个等待线程，避免惊群。

## 代码示例（Curator 框架）

```java
// Curator 封装了 ZooKeeper 分布式锁的所有细节
InterProcessMutex lock = new InterProcessMutex(client, "/locks/my_lock");

try {
    // 阻塞加锁
    lock.acquire();

    // 超时加锁
    if (lock.acquire(3, TimeUnit.SECONDS)) {
        try {
            // 业务逻辑
        } finally {
            lock.release();
        }
    }
} catch (Exception e) {
    // 处理异常
}
```

## 与 Redis 分布式锁对比

| 对比项 | Redis 分布式锁 | ZooKeeper 分布式锁 |
|--------|--------------|------------------|
| 性能 | 高 | 中（ZK 写操作需过半数节点确认） |
| 可靠性 | 中（主从切换可能丢锁） | 高（ZAB 协议强一致） |
| 死锁处理 | 依赖过期时间 | 临时节点自动删除 |
| 公平性 | 不保证 | 顺序节点保证公平 |
| 实现复杂度 | 低 | 中（推荐用 Curator） |
| 适用场景 | 高并发、对一致性要求一般 | 强一致性要求、金融场景 |

---

## 锁释放场景

| 场景 | 处理方式 |
|------|----------|
| 正常释放 | 主动删除临时节点 |
| 客户端崩溃 | Session 超时后临时节点自动删除 |
| 网络分区 | Session 超时后节点删除，其他客户端可获取锁 |

ZooKeeper 的临时节点机制天然解决了 Redis 分布式锁中「持锁进程崩溃后锁不释放」的问题，不需要额外的过期时间设计。
