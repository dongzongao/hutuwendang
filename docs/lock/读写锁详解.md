# 读写锁详解

## 为什么需要读写锁

普通互斥锁（synchronized / ReentrantLock）读读之间也互斥，但读操作不修改数据，读读并发是安全的。

读写锁允许：
- 多个线程同时读（共享）
- 写时独占，读写互斥，写写互斥

适合读多写少的场景。

---

## ReentrantReadWriteLock

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock  = rwLock.readLock();
Lock writeLock = rwLock.writeLock();

// 读操作
readLock.lock();
try {
    // 多线程可同时执行
} finally { readLock.unlock(); }

// 写操作
writeLock.lock();
try {
    // 独占，其他读写都阻塞
} finally { writeLock.unlock(); }
```

### 读写状态设计

AQS 的 state（32位）被拆分为两部分：

```
高 16 位：读锁持有数（共享）
低 16 位：写锁重入次数（独占）

state = 0x00030002
        ↑高16位=3（3个线程持有读锁）
                ↑低16位=2（写锁重入2次）
```

### 锁降级

写锁可以降级为读锁（持有写锁时获取读锁，再释放写锁）：

```java
writeLock.lock();
try {
    // 修改数据
    data = newValue;

    readLock.lock();   // 先获取读锁（降级）
} finally {
    writeLock.unlock(); // 再释放写锁
}
try {
    // 使用数据（此时持有读锁，其他写操作被阻塞）
} finally {
    readLock.unlock();
}
```

> 不支持锁升级（持有读锁时获取写锁会死锁）。

## StampedLock（JDK 8）

比 ReadWriteLock 更进一步，提供乐观读模式，读操作完全无锁。

### 三种模式

```java
StampedLock lock = new StampedLock();

// 1. 写锁（独占）
long stamp = lock.writeLock();
try { /* 写操作 */ } finally { lock.unlockWrite(stamp); }

// 2. 悲观读锁（共享，类似 ReadWriteLock）
long stamp = lock.readLock();
try { /* 读操作 */ } finally { lock.unlockRead(stamp); }

// 3. 乐观读（无锁，最高性能）
long stamp = lock.tryOptimisticRead();
// 读取数据到局部变量
int x = this.x, y = this.y;
// 验证期间是否有写操作
if (!lock.validate(stamp)) {
    // 有写操作，升级为悲观读锁
    stamp = lock.readLock();
    try { x = this.x; y = this.y; }
    finally { lock.unlockRead(stamp); }
}
// 使用 x, y
```

### 乐观读原理

`tryOptimisticRead()` 返回一个 stamp（版本号），读完后用 `validate(stamp)` 检查期间是否有写操作。没有写操作则直接使用，有则升级为悲观读锁重新读。

## 三种锁对比

| 对比项 | ReentrantLock | ReadWriteLock | StampedLock |
|--------|--------------|---------------|-------------|
| 读读并发 | ❌ | ✅ | ✅ |
| 乐观读 | ❌ | ❌ | ✅ |
| 可重入 | ✅ | ✅ | ❌ |
| 锁降级 | - | ✅ | ✅ |
| 条件变量 | ✅ | ✅（写锁） | ❌ |
| 适用场景 | 通用 | 读多写少 | 读极多写极少 |
